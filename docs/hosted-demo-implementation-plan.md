# Phase 6 — Hosted Demo Implementation Plan

> **Status:** Draft implementation plan.  
> **Project:** `miggalvez/officium-novum`  
> **Scope:** Hosted web demo / beta frontend, including reviewer-oriented “Report this” integration.  
> **Depends on:** Phase 4 read-only JSON API; Phase 5 reviewer-report contract.  
> **Recommended package:** `packages/web` as `@officium-novum/web`.

---

## 1. Purpose

The hosted demo is the first public, prayer-facing consumer of the Officium Novum API. Its job is not to invent liturgical logic, reformat the corpus, or become a second compositor. Its job is to let a real user or reviewer select a date, rubrical version, language, and Hour, then display the API result clearly, accessibly, and reproducibly.

The demo also provides the practical entry point for Phase 5 reviewer feedback. A reviewer who sees a disputed feast, commemoration, psalm, antiphon, lesson, rubric, translation issue, or rendering issue should be able to click **Report this**, submit a structured report, and give maintainers enough metadata to reproduce the exact output.

The central rule:

> The hosted demo is a consumer of the API, not a liturgical authority. The API remains the canonical composition boundary.

## 2. Current Repo Facts This Plan Assumes

The plan is based on the current repository architecture and Phase 4 API contract:

- The stack is already layered as Parser → Rubrical Engine → Compositor → API → Clients.
- Phase 4 is complete for the core read-only Breviary JSON API.
- The API exposes `/api/v1/status`, `/api/v1/versions`, `/api/v1/languages`, `/api/v1/office/{date}/{hour}`, `/api/v1/days/{date}`, `/api/v1/calendar/{year}/{month}`, `/api/v1/openapi.json`, and `/api/v1/docs`.
- The API uses canonical `version` handles, with `rubrics` aliases normalized immediately.
- Public language tags are `la` and `en`, mapped internally to corpus language names.
- The API already returns deterministic cache headers and ETags.
- The repository workspace currently includes `packages/*`, so a frontend package under `packages/web` will naturally participate in root `pnpm -r` commands.

## 3. Product Goals

### 3.1 Public demo goal

Let users pray or inspect the traditional Roman Breviary Office generated by the new stack.

Minimum successful user journey:

1. Open the hosted demo.
2. See today’s Office using a sensible default version and language.
3. Change date, Hour, rubrical version, and language.
4. Read the result in a stable, typographically clear layout.
5. Copy or share a permalink that reproduces the same request.
6. Open the raw API response or API documentation when needed.

### 3.2 Reviewer goal

Let qualified reviewers inspect outputs and submit actionable reports.

Minimum successful reviewer journey:

1. Navigate to the disputed date, Hour, version, and language.
2. Click **Report this**.
3. Confirm the exact request metadata captured by the demo.
4. Describe expected vs. actual behavior.
5. Add a citation or consultation note.
6. Submit through GitHub issue or email without the maintainer having to reconstruct the request by guesswork.

### 3.3 Maintainer goal

Let maintainers reproduce reports cheaply and turn accepted reports into fixtures.

Minimum successful maintainer journey:

1. Receive report with exact API path, frontend build SHA, API content version, requested date, Hour, version, language, and orthography.
2. Re-run the same request locally or against the hosted API.
3. Classify the report under the Phase 5 adjudication taxonomy.
4. Land a fixture in the owning package.
5. Update `REVIEWER_REPORTS.md` or equivalent public changelog.

## 5. Phase Boundary

This plan should be treated as **Phase 6: Hosted Demo / Frontend**, with one explicit bridge to Phase 5.

Phase 5 owns:

- reviewer-report schema;
- adjudication workflow;
- fixture lifecycle;
- public report changelog;
- privacy policy for reports;
- classification taxonomy.

Phase 6 owns:

- the hosted demo UI;
- the report button UI;
- client-side report payload generation;
- GitHub/email handoff;
- deployment of the web app;
- user-facing rendering, accessibility, and offline behavior.

The first report-button implementation should **not** add a general write endpoint to the API. The API is currently deliberately read-only. Keep it that way for v1. The clean first version is a frontend-driven GitHub/email handoff. A structured server-side report intake can be added later as a separate, reviewed scope.

## 7. Technology Recommendation

### 7.1 Frontend framework

Use **Vite + React + TypeScript** for `packages/web`.

Reasons:

- TypeScript matches the rest of the repo.
- Vite is enough; no server-side rendering is needed.
- React has the broadest contributor familiarity.
- The UI is component-heavy but not application-server-heavy.
- A static build keeps hosting cheap.

Acceptable alternatives: Svelte or Vue. Do not use Next.js for the first version unless there is a specific need for server features. The API already does the hard work; adding an SSR framework here would be like wearing a fiddleback chasuble to mow the lawn — impressive, but not the point.

### 7.2 Styling

Use plain CSS with CSS custom properties, or CSS modules.

Avoid heavy design systems in the first version. The frontend needs excellent typography, spacing, responsive columns, and accessibility. It does not need a cathedral of dependencies.

Recommended style files:

```text
src/styles/
  reset.css
  tokens.css
  typography.css
  layout.css
  office.css
  calendar.css
  forms.css
  print.css
```

### 7.3 State management

Use local component state plus a small settings store. Do not introduce Redux.

State categories:

- URL state: date, hour, version, languages, orthography.
- Local preference state: default version, default languages, font size, display mode.
- API response state: current office/day/calendar/status payloads.

For the first version, a small typed fetch layer is enough. If request caching becomes fiddly, introduce a query library later. The browser HTTP cache and service worker will already do some work because the API has deterministic cache headers.

### 7.4 Client DTO typing

Start with narrow public DTO types in `packages/web/src/api/types.ts`, copied from the API’s public DTO boundary.

Later improvement: generate types from `/api/v1/openapi.json` during CI or build.

Do **not** import private API internals into the web package. The frontend should consume the API as an external client would. That is the whole point of having a public contract.

## 10. Routing Plan

### 10.1 Public routes

```text
/                                         → redirect to today’s default Office
/office/:date/:hour                       → single Hour view
/day/:date                                → day overview with Hour navigation
/calendar/:year/:month                    → monthly calendar
/settings                                 → preferences
/status                                   → API/demo build status
/about                                    → project explanation and limitations
/api                                      → links to /api/v1/docs and /api/v1/openapi.json
```

### 10.2 Query parameters

Use the same public contract as the API where possible:

```text
version=Rubrics%201960%20-%201960
rubrics=1960                # optional inbound compatibility only; normalize in route state
lang=la,en
langfb=en
orthography=version
strict=true
mode=parallel|sequential
fontSize=normal|large|larger
```

Frontend canonical URLs should use `version`, not `rubrics`, once the version is known.

### 10.3 Default route behavior

`/` should redirect to:

```text
/office/<today>/lauds?version=Rubrics%201960%20-%201960&lang=la,en&orthography=version
```

The default Hour can be either:

- Lauds, for a stable first implementation; or
- the Hour closest to local time, if implemented carefully and transparently.

Recommendation: start with Lauds. Time-sensitive routing creates more edge cases than value in the first hosted demo.

## 11. API Client Plan

### 11.1 Client responsibilities

The web client should:

- Build valid API URLs.
- Fetch JSON.
- Surface API errors clearly.
- Preserve `meta.canonicalPath` from API responses.
- Preserve ETags through browser cache naturally.
- Never synthesize liturgical fields missing from the API.
- Never rewrite public DTOs into a new internal liturgical model.

### 11.2 Client functions

```ts
export async function getStatus(): Promise<StatusResponse>;
export async function getVersions(): Promise<VersionsResponse>;
export async function getLanguages(): Promise<LanguagesResponse>;
export async function getOfficeHour(input: OfficeHourRequest): Promise<OfficeHourResponse>;
export async function getOfficeDay(input: OfficeDayRequest): Promise<OfficeDayResponse>;
export async function getCalendarMonth(input: CalendarMonthRequest): Promise<CalendarMonthResponse>;
```

### 11.3 Error handling

Map API errors to user-facing states:

| API status | Demo behavior |
|---:|---|
| `400` | Show invalid input message; keep controls visible. |
| `422` | Show composition-error warning; offer `strict=false` retry only if appropriate. |
| `501` | Show unsupported/deferred version message with API hints. |
| `500` | Show server error; include report button and raw request metadata. |
| network failure | Show offline/error state; use service-worker cache when available. |

The error page should still offer:

- Copy request URL.
- Open raw API URL.
- Report this.

Bad output and failed output are both useful data.

## 12. Office Rendering Plan

### 12.1 Rendering principles

- Preserve the API section order exactly.
- Preserve the API line order exactly.
- Render text runs without `dangerouslySetInnerHTML`.
- Do not infer missing rubrics.
- Do not hide warnings.
- Make rubrics visually distinct and semantically identifiable.
- Support Latin-only, English-only, and Latin+English layouts.
- Make Matins long-form reading tolerable.

### 12.2 Components

```text
OfficePage
  ├─ OfficeHeader
  │   ├─ date/version/hour/language controls
  │   ├─ day summary
  │   ├─ warnings
  │   ├─ raw JSON link
  │   └─ report button
  ├─ HourNav
  └─ OfficeRenderer
      └─ SectionRenderer[]
          └─ LineRenderer[]
              └─ RunRenderer[]
```

### 12.3 Run rendering

| Run type | Rendering |
|---|---|
| `text` | Main text. |
| `rubric` | Red-toned or distinct rubric style; not color-only. |
| `citation` | Smaller citation style; optionally collapsible. |
| `unresolved-macro` | Warning chip; visible in reviewer mode. |
| `unresolved-formula` | Warning chip; visible in reviewer mode. |
| `unresolved-reference` | Warning chip; visible in reviewer mode. |

### 12.4 Parallel language layout

For `lang=la,en`, default to parallel columns on wide screens:

```text
┌──────────────────────────┬──────────────────────────┐
│ Latin                    │ English                  │
├──────────────────────────┼──────────────────────────┤
│ Deus, in adjutorium...   │ O God, come to my aid... │
└──────────────────────────┴──────────────────────────┘
```

On narrow screens, switch to stacked language blocks per line:

```text
Latin
Deus, in adjutorium...

English
O God, come to my aid...
```

Do not use a literal HTML table for every line unless accessibility testing confirms it behaves well. CSS grid with semantic grouping may be better.

### 12.5 Reviewer mode

Add a toggle:

```text
Show validation details
```

When enabled, show:

- section `type`;
- section `slot`;
- section `reference`;
- warnings;
- unresolved run markers;
- API canonical path;
- content version;
- raw JSON link.

Normal users should not be forced to read internal metadata. Reviewers should not have to beg for it.

## 13. Day View Plan

Route:

```text
/day/:date?version=...&lang=...&orthography=...
```

Use the API day endpoint:

```text
GET /api/v1/days/{date}?version=...&lang=...&hours=all
```

MVP day view:

- day title;
- temporal information;
- celebration;
- commemorations;
- warnings;
- links to all eight Hours;
- optional inline rendering of one selected Hour.

Avoid rendering all eight Hours inline by default. Matins alone can be long; all eight Hours on one first-load page may feel like a monastic avalanche.

Recommended behavior:

- Day page fetches day summary and maybe selected Hour list.
- User clicks Hour tab to navigate to `/office/:date/:hour`.
- Later enhancement: inline expand/collapse per Hour.


- commemoration count;

Navigation:

- Previous month.
- Next month.
- Today.
- Version selector.

## 16. Report Button Implementation

### 16.1 Scope decision

The first report button should be implemented in the frontend and should produce a structured report that can be submitted through:

1. a prefilled GitHub issue; or
2. email to a reviewer-report address; or
3. copy-to-clipboard YAML/JSON for manual submission.

It should not require a new backend write endpoint.

### 16.2 Button placement

Show **Report this** in these places:

- Office Hour header.
- API error state.
- Day summary header.
- Calendar day detail popover or selected day panel.

For Office Hour reports, the report should be anchored to the specific date + Hour + version + language.

### 16.3 Captured metadata

The button should capture:

```yaml
context:
  reportSchemaVersion: 1
  generatedAt: <ISO datetime>
  frontend:
    buildSha: <VITE_OFFICIUM_BUILD_SHA>
    buildDate: <VITE_OFFICIUM_BUILD_DATE>
    publicBaseUrl: <VITE_OFFICIUM_PUBLIC_BASE_URL>
    route: <current browser route>
  api:
    baseUrl: <VITE_OFFICIUM_API_BASE_URL>
    apiVersion: v1
    contentVersion: <response.meta.contentVersion or status.content.contentVersion>
    upstreamSha: <status.content.upstreamSha if available>
    canonicalPath: <response.meta.canonicalPath if available>
  request:
    date: <ISO date>
    hour: <HourName or null>
    version: <VersionHandle>
    languages: [la, en]
    langfb: <optional>
    orthography: source | version
    strict: true | false
    apiUrl: <exact API URL>
  response:
    kind: office-hour | office-day | calendar-month | error
    warningCodes: []
    quality: complete | partial | unknown
```

### 16.4 Reviewer-entered fields

The dialog should ask for:

```yaml
disagreement:
  scope: feast | commemoration | psalter | lesson | antiphon | hymn | versicle | oration | rubric | translation | rendering | api | other
  expected: <free text>
  actual: <free text>
  selectedText: <optional excerpt from page>

citation:
  sourceType: ordo | rubrical-book | breviary | corpus | adr | consultation | none
  sourceId: <optional>
  reference: <free text, e.g. "FSSP Ordo 2024, p. 47">
  excerpt: <optional short excerpt>

reviewer:
  publicAttribution: anonymous | name-ok
  name: <optional>
  contact: <optional>
  affiliation: <optional>
  qualification: <optional>

notes: <free text>
```

The UI must clearly say:

- GitHub issues are public.
- Email reports are private until redacted/processed.
- Names and contact info are optional.
- Public artifacts should reference reports by ID, not private contact details.

### 16.5 Generated report formats

Support both JSON and YAML output. YAML is easier to read in issue bodies. JSON is easier for tooling.

Example generated report:

```yaml
reportSchemaVersion: 1
generatedAt: 2026-04-28T18:30:00.000Z
frontend:
  buildSha: abc1234
  route: /office/2026-04-28/lauds?version=Rubrics%201960%20-%201960&lang=la,en
api:
  apiVersion: v1
  contentVersion: def5678
  canonicalPath: /api/v1/office/2026-04-28/lauds?version=Rubrics%201960%20-%201960&lang=la,en&orthography=version&strict=true
request:
  date: 2026-04-28
  hour: lauds
  version: Rubrics 1960 - 1960
  languages:
    - la
    - en
  orthography: version
  strict: true
disagreement:
  scope: commemoration
  expected: "..."
  actual: "..."
citation:
  sourceType: ordo
  reference: "..."
reviewer:
  publicAttribution: anonymous
notes: "..."
```

### 16.6 GitHub issue handoff

Add a GitHub issue template:

```text
.github/ISSUE_TEMPLATE/reviewer-report.yml
```

Template fields:

- Date.
- Version handle.
- Hour.
- Language.
- API canonical path.
- Scope.
- Expected.
- Actual.
- Citation source.
- Citation reference.
- Notes.
- Public attribution consent.

The frontend can open:

```text
https://github.com/miggalvez/officium-novum/issues/new?template=reviewer-report.yml&title=Reviewer%20report%3A%202026-04-28%20Lauds
```

Do not rely on URL query parameters for the full payload if it becomes too long. For longer payloads:

1. open the issue template;
2. copy the generated YAML to clipboard;
3. instruct the reviewer to paste it into the issue body.

### 16.7 Email handoff

Provide:

- `mailto:` link with concise subject and summary body;
- copy-to-clipboard full YAML;
- instruction to paste YAML into the email.

Subject format:

```text
Officium Novum reviewer report: 2026-04-28 lauds Rubrics 1960
```

### 16.8 Future server-side report intake

Only add this after the GitHub/email flow proves insufficient.

Possible future endpoint:

```text
POST /reports/v1/reviewer-reports
```

Do not put this under the existing read-only Office API unless the project intentionally expands the API’s responsibility. A separate report-intake service or serverless function is cleaner.

Server-side intake would need:

- spam protection;
- rate limiting;
- private data storage;
- redaction workflow;
- abuse handling;
- consent logging;
- export to fixture format.

That is not free. It is a real product surface, not a button with delusions of grandeur.

## 17. Offline and Caching Plan

### 17.1 Goals

- The app shell should load offline after first visit.
- Recently viewed Office responses should be readable offline.
- Users should be able to pre-cache the current week for their selected version/languages.
- API caching must respect deterministic request URLs and ETags.

### 17.2 Service worker strategy

Cache buckets:

```text
app-shell-v<buildSha>       # HTML, CSS, JS, manifest, icons
api-runtime-v1              # GET /api/v1/* responses
```

App shell:

- Precache built static assets.
- On new build, update app shell and notify user.

API runtime cache:

- Cache successful `GET` responses for `/api/v1/office`, `/api/v1/days`, `/api/v1/calendar`, `/api/v1/status`, `/api/v1/versions`, `/api/v1/languages`.
- Use cache-first when offline.
- Use network-first or stale-while-revalidate when online.
- Respect exact URL identity. Language order matters because `la,en` and `en,la` can be display-distinct.

### 17.3 “Cache this week” feature

Add a small action:

```text
Cache this week for offline use
```

For the selected settings, prefetch:

```text
7 dates × selected hours
```

Recommended default selected hours:

- Lauds;
- Vespers;
- Compline.

Full all-hour caching can be offered, but warn that Matins can be large.

### 17.4 Offline fallback

When offline and requested content is unavailable:

- show a clear offline message;
- show cached dates available;
- preserve controls;
- do not pretend the content exists.

## 18. Accessibility Plan

Accessibility is not polish here. People pray in poor light, on small screens, while tired, while traveling, and sometimes with impaired sight.

Requirements:

- Semantic landmarks: `header`, `main`, `nav`, `footer`.
- One clear `h1` per page.
- Keyboard-accessible controls.
- Skip-to-content link.
- Visible focus styles.
- High-contrast text.
- Rubrics not distinguished by color alone.
- `lang="la"` and `lang="en"` on language-specific blocks where practical.
- Screen-reader labels for date, Hour, version, and language controls.
- Responsive columns that do not scramble reading order.
- Reduced-motion support.
- Print stylesheet.

Testing:

- Manual keyboard pass.
- Screen-reader smoke test.
- Automated accessibility check in component or E2E tests if dependency weight is acceptable.

## 19. Security and Privacy Plan

### 19.1 Security

- Never render API text through `dangerouslySetInnerHTML`.
- Escape all displayed text by default through React rendering.
- Use a restrictive Content Security Policy in production.
- Use same-origin deployment where possible.
- If cross-origin, configure explicit CORS origins; do not use wildcard CORS for production casually.
- Do not expose private deployment secrets to `VITE_*` variables.
- Do not log full reviewer contact details in public artifacts.

### 19.2 Privacy

- No user accounts.
- No tracking pixels.
- No invasive analytics in the first hosted demo.
- No IP/fingerprint collection for report button v1.
- Contact information optional.
- GitHub issue submission must warn that GitHub issues are public.
- Email submission must warn that maintainers may later publish a redacted report ID/status.

### 19.3 Error reporting

Use client-side error boundaries and a copyable diagnostic block.

Avoid automatic third-party error tracking in v1. If later added, scrub:

- report form contents;
- contact details;
- free-text expected/actual fields;
- full API payloads unless explicitly allowed.

## 20. Deployment Plan

### 20.1 Preferred shape

Use two deployable units:

```text
1. API service       → existing packages/api Fastify server
2. Static frontend   → packages/web Vite build output
```

Preferred public routing:

```text
https://demo.example.org/              # frontend
https://demo.example.org/api/v1/...    # reverse-proxied API
```

Same-origin routing is easiest for users, browsers, and caching.

### 20.2 API deployment requirements

The API host must support:

- Node runtime compatible with the package.
- Access to the `upstream/web/www` corpus files.
- Startup command: `pnpm -C packages/api start` after build.
- Environment variables from §9.
- Health check against `/api/v1/status`.
- Stable content version set from commit SHA or release ID.

Recommended production API build flow:

```text
pnpm install --frozen-lockfile
pnpm -r build
OFFICIUM_API_HOST=0.0.0.0 pnpm -C packages/api start
```

If the repo does not yet have `build` scripts for every package, add them only where necessary and keep root scripts simple.

### 20.3 Frontend deployment requirements

Frontend host must support:

- static asset hosting;
- SPA fallback to `index.html`;
- HTTPS;
- custom response headers;
- cache headers for hashed assets;
- optional reverse proxy to API.

Build:

```text
pnpm -C packages/web build
```

Output:

```text
packages/web/dist/
```

### 20.4 Cache headers

Frontend static assets:

```http
Cache-Control: public, max-age=31536000, immutable
```

For `index.html`:

```http
Cache-Control: public, max-age=0, must-revalidate
```

API responses already emit deterministic cache headers and ETags. Do not override them blindly at the frontend host.

### 20.5 Preview deployments

Every frontend PR should produce a preview deployment when feasible.

Preview deployments should use either:

- a preview API; or
- production API with a clear banner saying the frontend is preview but data is production.

## 24. API Changes Recommended for Hosted Demo

Keep these small.

### 24.1 CORS or same-origin proxy

Preferred: same-origin reverse proxy.

If not possible, add CORS support to `packages/api`:

```text
@fastify/cors
```

Config:

```ts
OFFICIUM_ALLOWED_ORIGINS=https://demo.example.org,http://localhost:5173
```

Acceptance:

- Local frontend can call local API.
- Production frontend can call production API.
- Wildcard production CORS is avoided.

### 24.2 Status endpoint metadata

Ensure `/api/v1/status` exposes enough for reports:

```ts
content: {
  contentVersion: string;
  upstreamSha?: string;
  corpusFileCount?: number;
}
```

If `upstreamSha` is not currently available, add it when practical. It is useful for reviewer reports because corpus changes matter.

### 24.3 No report POST endpoint in v1

Do not add `POST /api/v1/reports` as part of this first hosted demo. That expands the API from read-only public data into user-submitted content, with privacy and abuse obligations.

## 26. Risks and Mitigations

| Risk | Mitigation |
|---|---|
| Frontend accidentally reimplements liturgical logic | Treat API DTO as source; no client-side feast/rubric inference. |
| API read-only boundary gets blurred by report intake | Use GitHub/email handoff first; defer server-side intake. |
| Report payload leaks private reviewer data | Make private fields optional; warn about GitHub publicity; tests for redaction. |
| Matins rendering becomes unwieldy | Add section anchors, nocturn/lesson headings, collapse options after MVP if needed. |
| Offline cache serves stale app shell | Version app-shell cache by frontend build SHA. |
| Users mistake demo for final replacement | Add beta banner and clear project/about page. |
| Calendar view invents day color | Render only fields exposed by API. No vestment roulette. |
| CORS/deployment complexity | Prefer same-origin reverse proxy. |
| Heavy dependencies bloat static app | Start with Vite/React/plain CSS; add libraries only when pain is proven. |

## 29. Bottom Line

Build the hosted demo as a disciplined API consumer:

```text
API DTO → typed client → accessible renderer → permalink → reviewer report payload
```

Keep the frontend modest, typographically strong, and brutally honest about what it does. The project has already done the hard liturgical work in the parser, engine, compositor, and API. The hosted demo should make that work usable and reviewable without smuggling in a second implementation under a pretty coat of CSS.
