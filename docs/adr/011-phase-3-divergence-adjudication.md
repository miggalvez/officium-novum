# ADR-011: Phase 3 divergence adjudication protocol and sidecar schema

- **Status:** accepted
- **Date:** 2026-04-18
- **Deciders:** Phase 3 composition engine sign-off
- **Related:** `docs/phase-2-rubrical-engine-design.md` §19.4,
  `docs/rubrical-sources.md`,
  `packages/compositor/test/fixtures/compare-phase-3-perl.mjs`,
  `packages/compositor/test/divergence/adjudications.json`,
  ADR-010

## Context

Phase 3 ships with a live Perl comparison harness
(`packages/compositor/test/fixtures/compare-phase-3-perl.mjs`). On each run it
composes every hour for a matrix of dates under every configured rubrical
policy, runs the legacy Divinum Officium Perl renderer on the same inputs, and
emits a per-policy divergence ledger at
`packages/compositor/test/divergence/<policy>-2024.md`.

The ledger currently lists every divergent hour in a flat table. It does not
record whether a given divergence reflects an engine bug, a Perl bug, an
ambiguous rubrical question, or a purely representational difference. Without
that classification:

1. The <10-unresolved-rows-per-policy-per-year exit criterion inherited from
   Phase 2 §22.9 is not measurable.
2. Reviewers cannot distinguish "work remains" from "work completed; the rest
   are filed upstream bugs or documented ambiguities."
3. Contributors cannot resume the burn-down without re-doing prior review.

The Phase 2 engine reuses a comparable adjudication protocol (design §19.4):
Ordo Recitandi is primary, governing rubrical books are secondary, Perl output
is a tertiary comparison target and is never an oracle. Phase 3 inherits that
authority order but needs its own mechanics because the compare harness
rewrites the ledger `.md` files on every run. Any classification or citation
stored directly in the table would be erased the next time the harness ran.

## Options Considered

### Option A — In-ledger Class/Citation columns, hand-edited

*Each row in the generated `<policy>-2024.md` ledger carries `Class` and
`Citation` columns filled in by hand.*

- Pro: Readable — everything lives in one file.
- Con: The ledger is regenerated by
  `compare-phase-3-perl.mjs:renderDivergenceDoc`. Hand edits are lost on
  every run. Workable only if the harness is rewritten to *merge* existing
  rows — which requires a stable key anyway.

### Option B — Sidecar file keyed by stable row identity, merged at render time

*Adjudications live in `adjudications.json`, a hand-maintained sidecar. The
harness reads it, never writes it, and merges `Class` + `Citation` into the
generated ledger at render time.*

- Pro: Survives ledger regeneration.
- Pro: Machine-parseable for CI gates, stats, and bulk queries.
- Pro: Clear separation between auto (ledger `.md`) and hand (sidecar)
  artefacts.
- Con: Two files to scan instead of one.

### Option C — Classifications embedded in `docs/adr/` entries only

*No sidecar; every adjudicated row triggers an ADR that cites its own row
key.*

- Pro: Everything becomes a first-class architectural record.
- Con: Scales poorly. Even if 90 % of rows collapse to
  `rendering-difference` citing a single corpus line, writing 90 ADRs is not
  useful; they belong in one sidecar entry pointing at the shared citation.

## Decision

**Option B.** Adjudications live in
`packages/compositor/test/divergence/adjudications.json`. The harness loads
this file, never writes it, and renders a `Class` + `Citation` column in each
generated ledger by looking each row's key up in the sidecar. Rows without a
matching key render as `unadjudicated`.

### Classification taxonomy

Every adjudicated row carries exactly one class:

- `engine-bug` — the compositor is wrong. Fix it, add a regression test, and
  the row will collapse on the next harness run. Citation points at the Ordo
  or rubrical book that establishes the expected behaviour.
- `perl-bug` — the compositor matches the primary source; the legacy Perl
  renderer is wrong. Append a short entry to `docs/upstream-issues.md` so the
  upstream maintainers can triage it. Citation points at the source the
  compositor agrees with.
- `ordo-ambiguous` — neither the Ordo nor the governing book is dispositive.
  Write an ADR documenting the interpretation chosen and why. Citation points
  at that ADR.
- `rendering-difference` — purely surface (whitespace, marker prefix, quote
  style) and not liturgically material. Citation points at the normaliser
  location in `compare-phase-3-perl.mjs` or the corpus line that backs the
  retained form. Use sparingly; if a rubrical book addresses the
  representation, the class is `ordo-ambiguous` with an ADR, not
  `rendering-difference`.

**A divergence is never resolved by "matching the Perl" alone.** Every
adjudication carries a citation from an Ordo Recitandi, a governing rubrical
book (Rubricae Generales 1911, Cum Nostra 1955, Rubricarum Instructum 1960),
an in-repo ADR, or the corpus `.txt` file itself when dispositive.

### Sidecar key schema

Each row's key is computed by `computeRowKey()` in the harness:

```
<policy>/<date>/<hour>/<hash>
```

where

- `policy` is the full `VersionHandle` as used on the command line
  (`"Rubrics 1960 - 1960"`, etc.),
- `date` is the ISO date (`"2024-01-01"`),
- `hour` is the Hour label (`"Matins"`, `"Lauds"`, …),
- `hash` is the first eight hex characters of the SHA-256 of
  `<normalized-first-expected>\0<normalized-first-actual>` where
  normalisation collapses whitespace and trims ends.

The hash captures the divergence *pattern*, not a bare line index, which means:

- Rows where Perl and the compositor produce the same first-divergence lines
  on different dates share structure but not key — classify each date
  separately, or fold many rows into a single entry using a shared
  `citation` + `notes` pair.
- If a compositor change alters the first-divergence line, the key changes,
  and the old entry surfaces as stale in the next ledger run (it will not
  appear anywhere) while the new pattern shows up as `unadjudicated`. This is
  the intended behaviour: the row has changed meaning and needs fresh review.

### Sidecar value schema

```json
{
  "Rubrics 1960 - 1960/2024-01-01/Matins/abcd1234": {
    "policy": "Rubrics 1960 - 1960",
    "date": "2024-01-01",
    "hour": "Matins",
    "firstExpected": "Ant. Christus natus est nobis: * Veníte, adorémus.",
    "firstActual": "Christus natus est nobis: * Veníte, adorémus.",
    "class": "engine-bug",
    "citation": "packages/compositor/src/emit/sections.ts — antiphon `Ant.` marker not emitted; source e.g. upstream/.../Sancti/01-01.txt:NN",
    "notes": "Covered by plan sub-phase 3c item 1.",
    "resolvedAt": "2026-04-25",
    "commitSha": "abcdef1"
  }
}
```

Only `class` and `citation` are required. The other fields are advisory but
strongly recommended. `firstExpected` and `firstActual` are duplicated into
the value to make the entry human-scannable without recomputing the hash.

## Consequences

- **Positive:** Classifications and citations survive harness reruns.
  Contributors can resume the burn-down without redoing review.
- **Positive:** The harness writes a machine-readable class breakdown into
  every ledger (`unadjudicated / engine-bug / perl-bug / ordo-ambiguous /
  rendering-difference` counts). This is the measurable surface for the Phase
  3 §22 success criterion of <10 unresolved rows per policy per year.
- **Positive:** CI can gate on `unadjudicated < N` per policy without parsing
  Markdown.
- **Negative:** Two artefacts per policy (the generated `.md` and the shared
  `adjudications.json`) instead of one. Mitigated by the sidecar being a flat
  JSON object and by the ledger's notes pointing at this ADR.
- **Negative:** When the compositor changes in a way that alters first-
  divergence lines, old adjudication entries become orphans. They remain in
  the sidecar but no longer apply to any row. This is acceptable — they
  serve as an audit trail — but a follow-up housekeeping pass every few
  sprints should prune orphans.
- **Follow-up:** Introduce `ADJUDICATION_LOG.md` (per the completion plan,
  sub-phase 3h) as a chronological human-readable audit log keyed by commit
  SHA. The log is append-only and complements, not replaces, the sidecar.
- **Revisit trigger:** If Phase 4 needs to surface adjudication data through
  the API (e.g., annotating a composed Hour with "this section has an open
  divergence"), migrate the sidecar into a typed, queryable artefact rather
  than a free-form JSON object.

## Notes

The protocol inherits CLAUDE.md §19.4 verbatim:

> When the engine diverges from Perl, follow design §19.4: consult the Ordo,
> cite the resolution in `test/divergence/<policy>-2024.md`, and either fix
> the engine or file the Perl bug upstream. Never resolve a divergence by
> "matching the Perl" alone.

The sidecar is that "cite the resolution" step made durable. The ledger `.md`
rows remain the discoverable index; `adjudications.json` is the authoritative
record of what has been decided.
